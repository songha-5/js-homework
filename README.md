- 첫번째 문제에서 한두시간 정도 코드를 째려보기만 했습니다. boolean을 사용하면 될거같은데 막상 짜보려니 어떻게 짜야할지몰라서 결국 검색하는데 30분을 추가로 사용했습니다.

- 이상하네요. 코드가 됬다가 안됬다가 합니다. 분명히 똑같은 조건식을 적었고 잘 안돌아가서  30분동안 고생하다 결국 다른 코드로 바꾸었습니다. 결국 안되서 다시 그 코드를 보기위해 주석 을 풀었더니 되었습니다.

- 비밀번호 유효성 검사는 앞의 문제보다 금방 풀었습니다. 다만 이 코드도 password라는 중복 변수가 있는데 이걸 더 줄일 방법을 찾아보아야 겠다는 생각이 듭니다.

- 4번문제인 포인트 적립 계산을 다 엎었습니다. 처음에는 논리연산자로 풀려고 했는데 너무 조건문이 길어지고 아무리해도 결과값이 안나오고 중간에 값도 이상하게 나오길래 노트를 뒤져봤더니 해당 문제에서 쓰는게 아닌거 같습니다. 간단히 true, false로 나뉘는 값에서 사용되는거고 여러 등급이나 구분이 있는 경우엔 조건을 달아서 써야함으로 맞지 않은걸 직접 배웠습니다.

- 4번 문제에 객체 접근에서 꽤나 골치가 아팠습니다. 바로 객체를 떠올리지 못해서 꽤나 돌아간 탓도 있었지만, 큰 문제는 접근법이였습니다. 객체에서 괄호로 접근해야 변수로 값을 받아 key값과 동일한 값을 적용할 수 있었는데 . 표기법을 사용하여 꽤나 헤메였습니다.

- 5번 문제의 시간을 직접 10으로 적었는데 이걸 보다 10:00으로 적을 수 없을까해서 찾아봤었습니다. 하지만 너무 배우지 못한 부분이여서 넘어갔습니다. 잘 알지도 못하는 상태에서 쓰는것은 아니라 생각했습니다. 제가 이해해서 쓰는것이 아닌 배끼는 정도로 사용할 것 같았습니다. 차라리 선생님께 제대로 배우고 사용하는것이 낫다고 생각하여, 이런 코드가 있다는것만 알아가고 넘어갔습니다 (date())

- 코드 작성 전에 한글로 어떻게 짤건지 리스트를 작성 후 코드를 작성했습니다. 일단 지금까지 코드를 작성한 방법중 가장 괜찮은거 같습니다.

- 2번 문제를 다시 푸는중에 왜 논리 연산자가 제대로 작동 안하는지 알았습니다. 검색을 해보니 &&와 ||가 같이 쓰였을 때, &&먼저 계산하고 ||를 계산한다고 합니다. 나는 왼쪽에서 오른쪽으로 읽었으니 이해가 안됬던 것입니다.

- 2번 문제에서 수정 전 코드는 일반 지역의 기본배송비 3_000원과 추가 배송비 3_000원이 같으니 같은 const로 한번에 선언해 묶었었습니다. 그런데 생각해보니 그렇게 하면 안된다는것을 알았습니다. 추가 배송비가 4_000이 된다면 완전 새롭게 코드 리펙토링을 해야하는 경우가 생긴다는 것을 깨달았습니다. 전 코드에서 다시 수정해 반영하였습니다.

- 2번 문제 다시 푸는 와중에 의아한 일이 생겼습니다. 값이 음수일때 0이 뜨게 해야하는데 ``const result = price <= 0 && 0 || totalPrice``이렇게 값을 입력하니 0이 falsy한 값이라 0이 입력 안되서 '0'이렇게 문자열로 넣으려고 하니 ''빈문자열일때 NaN오류가 나 문제가 생겼습니다. 그러다가 아무 생각없이 ||연산자 앞뒤를 바꾸게 됬고 ``const result = price > 0 && totalPrice || 0`` 이렇게 하니까 문제없이 작동되었습니다.

# - 과제 코드 리뷰
- welcomeMessage 함수

  - welcomeMessage() 코드에서 ``"return (isVip && vipWelcome) || (isVip || nomalWelcome)"`` 이 코드가 읽기 너무 불편하고 한눈에 알아보기 힘들었다. 
  
  - 실제로 ``"(isVip || nomalWelcome)"`` 이 코드에서 ||을 &&로 썼는데 작동이 안되서 꽤나 고생했다. 알고보니 &&연산자는 isVip값이 false면 더이상 연산을 돌리지 않기 때문에 뒤의 welcome값이 안보였던 것이였다.
  
  - isVip가 중복으로 들어가 해당 코드를 너무 읽기 힘들다. 물론 중간에 ||이 있어 두개 다른 연산자로 묶여있다는걸 알 수 있어 좋지만 뭔가 미묘하게 복잡하다고 느껴진다.

- deliveryCalculate 함수
  - 배열을 하나씩 const를 만들어서 제주, 도서로 만드니 쓰기도, 관리하기도 힘들었다 더 나은 방법이 없는지 알아봐야겠다.

  - 5문제중에 가장 코드를 알아보기 힘든거 같다. 코드 리펙토링이 필요해 보인다. 특히 비교 연산자가 여러개 걸려있으면서 해당 변수를 변수로 나눠서 정리하지 않은탓에 너무 조건이 길게 보여 한눈에 보이지가 않는다.

  - '', '-마이너스', '0'이 처리가 안된다. 해당 조건을 따로 걸어서 처리하지도 못할 뿐더러 코드가 너무 길고 복잡해 따로 조건을 추가하기도 힘들다. 코드를 엎고 새로 짜야한다고 생각한다.

  - 논리 연산자가 너무 복잡해서 새로 짰습니다. 이해하지 못한 상태에서 짠 코드라 코드 정리도 안되었고 너무 복잡해서 다시 짜야한다고 생각해 새로 짰다.

  - 코드를 새로 짰다. 일단 일반지역과 추가배송비가 같은 3_000원 변수를 사용했는데 일반 지역의 값이 바뀌는 경우 다시 코드를 짜야한다는것을 깨달았다.

  - 배송비 값이 -1일때 0을 반환해야하는 코드를 짰다. 처음에 0이 falsy한 값이라 실패를 했는데, 어쩌다보니 우연찮게 값의 순서를 바꾸어서 잘 작동하게 됬다. 아마 추리를 하자면 ``const result = price <= 0 && 0 || totalPrice``에서 ``price <= 0 && 0``가 price가 -1이 들어와도 false가 됬다. 왜냐하면 0은 falsy한 값이라 이 값전체는 false가 price가 맞게 들어와도 false만 된다. 그런데 마지막 || 0은 왜 되는지 모르겠다 아마 마지막값이라 그런가 싶다.


- passwordValid 함수

  - 변수가 하나 사용되었으므로 ()를 생략하고자 했다.

  - password가 중복으로 사용되서 아쉽다. 처음에는 ``8 <= password <= 20``로 썼지만 되지않는걸 보니 논리 연산자는 무조건 조건을 나눠야하는거같다.

- pointsCalculate 함수

  - 배열을 못떠올리고 논리연산자로 어떻게든 처리하려다가 실패한 코드다 논리 연산자에 대해 좀 더 알수 있었다.

  - []와 .표기법에 대해 알 수 있는 코드였다. '.표기법은 상수명 자체를 값으로 인식'한다. 즉 ```userRating.membership```을 부르면  객체 ```const=userRating```의 key값에서 'membership'을 찾는다 라는 듯이다. 
  
  - return값을 number로 반환하라고 하셨으니 마지막에 parseInt를 추가했다

- movieTicket 함수

  - 조조할인 공식 계산 (시간이 10시 이전인지 확인, 20% 할인 계산식 추가)

  - 총 금액의 값을 구한다. 할인 시간이면 할인 금액으로 결제해주고 아니면 원래 금액으로 결제

  - 이번 코드는 가장 시간이 적게들었고 (화살표 함수 제외) 오류 없이 순서대로 작업했던 코드다.

  - 배열을 제대로 적용하니 여러 타입값을 쉽게 넣고 불러올 수 있어 편했으며, 논리 연산자를 첫번째 문제보단 이해하고 사용해서 두번째문제보다 읽기 쉬운 코드를 작성한거 같다.


# - 회고 (깨달은 점, 아쉬운 점 등)
- 첫 문제를 풀때, 결국 첫문제를 온전한 내 힘으로 풀지 못하고 검색을 사용해서 풀었다는 점에서 많이 아쉬웠다. 이 지식이 내 지식이 되려면 더 시간이 필요할거같다는 생각이 들고, 과제제출이 끝나면 이 문제랑 비슷한 문제를 풀어봐야겠다는 생각이 들었다.

- 배송비 계산에서 내가 짠 코드임에도 불구하고 너무 복잡하다고 느꼈다. 특히 배송비 추가 로직이 복잡하다 느껴졌다. 이건 논리 연산자가 잘 이해하지 못했단 뜻이고, 내가 너무 어렵게 짠 뜻이기도 하다. 이 부분을 코드리펙토링 하고싶은데 어디서 부터 손대야 할지 모르겠고, ''빈배열 추가나 0, -1일때 수정을 하고싶은데 아예 감이 안잡혀 다른 코드 부터 손대고 마지막에 손대기로 했다. 무언가, 어렴풋이 이런걸 활용해서 하면 되지않을까 생각은 드는데 (논리 연산자, 비교 연산자를 짜서 하면 되지않을까) 직접 코드를 짜려고하니 활용을 못해 충격적이다. 직접 코드를 짜는건 다른일이구나 싶어서 일주일에 시험을 볼 수 있는 기회가 있다는게 다행이라고 생각한다. 물론 내 점수는 좋지 않았지만 그게 오히려 좋다고 생각한다. 그리고 좀 아쉽다고 느껴지는건 시간은 한정되어있고 공부할 시간이 너무 부족하다는 점이다. 이번 주말에 과제 풀 시간도 너무 적고, (문제수를 줄여 달라는 것은 아닙니다) 따로 CS공부를 하고있는데 이것도 공부할 시간이 부족하다는 점이다. 고민인 부분은 복습할때 노션에 노트정리를 하고 그것을 필사로 한번 더 정리를 하는데 그렇게 하니 잘 외워지기는 하지만 예습을 할 시간은 없다는 점이다. 어떻게 해야할지 고민이다.

- 계속 썼던 코드로만 해결하려고하고(무조건 논리, 비교연산자만 쓰려고 한다), 많이 작성한 코드를 지우고 새롭게 코드를 짜거나 다른 방식으로 접근하려고 하지 않는 모습을 발견했다. 긍정적으로 생각하면 지금까지 알고있는 스펙트럼이 좁으니 그렇다, 더많은 해결 방법을 알면 괜찮지 않을까 생각하지만 한편으로는 알고있다. 문제 접근 방식을 다양하게 접근하지 않고 있다는 점을 알아버렸으니 좀 더 성실하게 코드작성에 임해야겠다는 생각이 들었다.

- 객체에서 괄호표기법을 어떻게 써야하는지 몸으로 하나 배웠다. 이론상으로 select같이 "유저"가 선택한 값에서만 괄호 표기법을 사용하고, 이름이 공백이나 하이픈에 추천한다 라고만 일았는데 . 표기법을 사용하니 상수로 넣은 값이 선택이 안되는것이다. 직접 써보고 안 점은. 이론으로 배웠던 . '표기법은 상수명 자체를 속성 이름으로 인식'으로 한다 라는 말이 이뜻인거 같다. ``const userRating = { 'vip': 5 }``가 있으면 .으로 불러올 때 안되니까, ``const vip = {'vip : 5}``라고 값과 동일한 vip라는 상수명으로 동일하게 넣으면 작동은 하는데, 이렇게 바뀌는 값인데, 고정된 상수명을 사용할 수 없으니(이러한 변수명은 지양해야한다, 다른 변수명이 들어오면 바로 에러인데다가 회원등급이라는 적절한 상수명이 들어가야 맞다) 밖에서 받아오는 membership을 사용해야하고, .표기법은 key값인 vip를 안찾고 membership이라는 키값을 찾으니(userRating에는 'membership'이라는 키값이 없어!) 없다고 뜨는것이였다. 세상에 나는 이 부분이 너무 흥미로워서 잊지않고있어야지 라고 생각하고 있었는데, 딱 HTML ``<select>``부분만 적용할 줄 알았고, 직접 다른 케이스에는 적용을 못했던 것이였다. 이번에 하나 배웠으니 다음에는 보다 빨리 이 부분을 해결할 수 있고, 언젠간 TDZ처럼 보기도 전에 '음 이건 문제가 일어나는 코드야'라고 알 수 있을 정도가 되었으면 좋겠다.

- 5번 문제를 풀고있는데 이제야 어떻게 짜야하는지 1/10이나마 감이 잡힌다. 한글로 어떤 코드를 짤건지 순서대로 작성하고 (1. 받아오는 데이터 예시 - 'IMAX 영화', 2, 10 / 2. 영화 리스트를 담아와서 정리해야함 / 3. 담아와서 정리해둔 값을 인원이랑 계산해야함 / 4. 계산한걸 할인해야함 / 4-1. 할인하려면 조조 시간으로 구분해야함 (변수 앞에 넣어야함) ... ) 그거에 따라 일단 코드를 작성해보고 오류가 나면 다시 생각으로 돌아가고있다. 아이러니하게도 1-5번중에 가장 순조롭게 풀리는것은 5번이였다. 물론 코드의 퀄리티는 어떻게 될지는 모르겠다. 시간이 허락한다면 앞의 문제도 다시 풀어봐야겠다

- 논리 연산자의 &&와 ||이 같이 쓰였을 때 우선순위가 있는지 몰랐습니다. 그래서 여러개를 썼을때 원하는 값이 나오지 않아 의아했었는데 왼쪽에서 오른쪽으로 읽는게 아니라 &&가 먼저 계산되고 ||이 계산되는걸 알았습니다. 이 부분을 배웠으니 응용을해서 제것으로 만들으려 노력해야겠습니다. 머릿속에선 이해가 되는거같은데 직접 사용하면 또 실수가 날거같아 걱정입니다.
